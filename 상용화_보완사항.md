## 1. 총평

현재 시스템은 기능 흐름(예약 생성 → 관리자 조회 → 상태 변경)은 완성되어 있으나, **보안·알림·결제** 등 실제 운영에 필수적인 요소가 빠져 있어 현재 상태로 상용 오픈은 권장하지 않습니다.

| 구분 | 현황 |
|------|------|
| 핵심 예약 흐름 | 완성 |
| 관리자 기능 | 기본 수준 완성 |
| 보안 (인증/인가) | 미구현 |
| 알림 (문자/알림톡) | 미구현 |
| 결제 | 미구현 |
| 운영 로깅/모니터링 | 미구현 |

## 2. 긴급 — 오픈 불가 수준

> 아래 항목이 해결되지 않으면 서비스 오픈 시 보안 사고 또는 데이터 사고가 발생할 수 있습니다.

---

### 2-1. 관리자 페이지 인증 없음

**현황**
`/admin`, `/admin/trainers` URL만 알면 누구든 접근 가능합니다. 별도의 로그인, 비밀번호, 세션 확인이 전혀 없습니다.

**위험**
- 외부인이 전체 예약 목록(회원 이름, 전화번호) 열람 가능
- 외부인이 예약을 무단으로 취소하거나 상태 변경 가능
- 트레이너를 무단으로 등록·수정 가능

**필요한 조치**
- 관리자 로그인 기능 개발 (ID/PW 또는 간편 PIN)
- 로그인하지 않은 상태에서 `/admin` 접근 시 로그인 페이지로 강제 이동
- 관리자 전용 API 엔드포인트에 인증 미들웨어 적용

---

### 2-2. AWS 자격증명 코드 내 평문 저장

**현황**
`backend/.env` 파일에 실제 AWS Access Key와 Secret Key가 평문으로 저장되어 있습니다.

**위험**
- 해당 파일이 GitHub 등 외부 저장소에 업로드될 경우 AWS 계정 전체가 탈취될 수 있습니다.
- 탈취된 키로 AWS 리소스 무단 사용 시 수백만 원 이상의 비용이 발생할 수 있습니다.

**필요한 조치**
- `.env` 파일을 `.gitignore`에 반드시 추가
- Lambda 환경에서는 AWS IAM Role 사용 (별도 키 불필요)
- 로컬 개발 환경에서는 AWS CLI 프로파일 방식 사용
- 현재 사용 중인 Access Key는 즉시 비활성화 후 재발급

---

### 2-3. CORS 전체 도메인 허용

**현황**
```
모든 출처(도메인)에서 API 호출 허용 상태
```

**위험**
- 어떤 외부 사이트에서도 이 서비스의 API를 직접 호출할 수 있습니다.
- 자동화 스크립트로 대량 예약 생성 또는 데이터 수집이 가능합니다.

**필요한 조치**
- 실제 서비스 도메인만 허용하도록 CORS 설정 변경
- 예: `https://gympt.co.kr`만 허용

---

### 2-4. 동시 예약 시 중복 발생 가능 (레이스 컨디션)

**현황**
예약 생성 시 "이 시간에 이미 예약 있는지 확인 → 없으면 저장" 순서로 처리합니다.
두 명이 동시에 같은 시간을 예약하면 둘 다 "확인" 통과 후 동시에 저장될 수 있습니다.

**위험**
- 같은 트레이너·날짜·시간에 2명 이상이 예약되는 중복 예약 발생

**필요한 조치**
- DynamoDB 조건부 쓰기(Conditional Write) 적용
- "저장 시점"에 중복 여부를 원자적으로 검증하도록 변경

---

### 2-5. 전화번호·날짜 유효성 검증 없음

**현황**
- 전화번호: 공백·하이픈만 제거하고 자리수 검증이 없어 `1`, `abc` 같은 값도 저장됩니다.
- 날짜: 과거 날짜나 10년 후 날짜도 예약이 가능합니다.

**필요한 조치**
- 전화번호: 10~11자리 숫자 형식 검증 추가
- 날짜: 오늘 이후 ~ 최대 3개월 이내만 선택 가능하도록 제한

## 3. 높음 — 오픈 전 필수 개발

> 서비스의 기본적인 신뢰성과 사용성을 위해 오픈 전 개발을 강력히 권장합니다.

---

### 3-1. 예약 확정 알림 없음

**현황**
예약이 완료되어도 회원에게 어떠한 연락도 가지 않습니다.
회원은 예약 완료 화면을 캡처해두거나 기억해야 합니다.

**영향**
- 회원이 예약 날짜·시간을 잊어버려 노쇼 발생 가능성 높음
- 예약 신뢰도 하락

**필요한 조치**
- 예약 완료 시 카카오 알림톡 또는 문자(SMS) 발송
- 발송 내용: 트레이너명, 날짜, 시간, PT 유형
- (선택) 하루 전 예약 리마인드 알림 발송

---

### 3-2. 회원이 직접 예약 취소 불가

**현황**
예약 취소는 관리자 페이지에서만 처리 가능합니다.
회원이 취소하려면 헬스장에 직접 연락해야 합니다.

**영향**
- 영업시간 외 취소 요청 처리 불가
- 직원 응대 부담 증가

**필요한 조치**
- "내 예약 조회" 페이지에서 예약 취소 기능 추가
- 취소 가능 기준 시간 설정 (예: 수업 2시간 전까지만 취소 가능)

---

### 3-3. 결제 기능 없음

**현황**
트레이너별 가격 정보는 저장되어 있으나 실제 결제 수단이 없습니다.
예약 후 현장 결제 또는 별도 연락을 통해 처리해야 합니다.

**필요한 조치** (선택 사항, 비즈니스 정책에 따라 결정)
- 온라인 결제 연동: 토스페이먼츠, 아임포트(포트원) 등
- 또는 현장 결제 안내 문구만 추가

---

### 3-4. 예약 이력 기록 없음

**현황**
예약을 취소하면 데이터베이스에서 완전히 삭제됩니다.
누가, 언제, 어떤 이유로 취소했는지 기록이 남지 않습니다.

**영향**
- 회원과 분쟁 시 근거 자료 없음
- 취소율, 노쇼율 통계 산출 불가

**필요한 조치**
- 예약 삭제 방식 → 상태를 'Cancelled'로 변경하는 방식으로 전환
- 변경 일시(`updated_at`), 변경 사유(`cancel_reason`) 필드 추가

---

### 3-5. 트레이너 정보 수정·삭제 불가

**현황**
트레이너 등록 기능만 있고, 등록 후 정보 수정이나 삭제가 불가합니다.
가격·경력·전문분야를 변경하려면 개발자에게 직접 요청해야 합니다.

**필요한 조치**
- 관리자 페이지에 트레이너 수정 기능 추가
- 관리자 페이지에 트레이너 비활성화(숨김) 기능 추가

## 4. 중간 — 운영 품질 개선

> 오픈 후 운영하면서 순차적으로 개선이 필요한 항목입니다.

---

### 4-1. 통계 기능 부족

**현황**
관리자 페이지에 오늘/이번주/전체/취소 4가지 수치만 표시됩니다.

**필요한 조치**
- 트레이너별 예약 현황 및 수익
- 월별/주별 예약 추이 그래프
- 시간대별 인기 슬롯 분석
- 노쇼율·취소율 통계

---

### 4-2. 오류 알림 방식 개선

**현황**
관리자 페이지에서 오류 발생 시 브라우저 기본 `alert()` 팝업이 표시됩니다.

**필요한 조치**
- 화면 내 토스트 메시지 또는 인라인 오류 메시지로 교체
- 오류 유형에 따른 안내 문구 세분화

---

### 4-3. 예약 완료 페이지 새로고침 시 데이터 소실

**현황**
예약 완료 후 이동하는 `/complete` 페이지는 이전 페이지의 상태 데이터에 의존합니다.
브라우저 새로고침 또는 직접 URL 접근 시 예약 정보가 표시되지 않습니다.

**필요한 조치**
- 예약 ID를 URL 파라미터로 전달하여 페이지 진입 시 서버에서 직접 조회

---

### 4-4. 서버 로깅 및 모니터링 부재

**현황**
서버에서 `console.error()`만 사용 중입니다.
운영 환경에서 오류가 발생해도 실시간 확인이 어렵습니다.

**필요한 조치**
- AWS CloudWatch 연동 (Lambda 사용 시 자동 수집 가능)
- 주요 API 호출 로그 저장 (언제, 어떤 요청, 성공/실패)
- 오류 발생 시 담당자 이메일·슬랙 알림 설정

---

### 4-5. 등록된 트레이너 상세 정보 불일치

**현황**
코드 내 기본 트레이너 데이터(Mike, Anna, Chris)에는 자격증·프로그램·소개 문구 등 상세 정보가 있으나, 관리자 페이지에서 신규 등록한 트레이너는 이름·전문분야·경력·가격·이미지만 저장됩니다.

**영향**
- 신규 등록 트레이너의 상세 프로필 페이지가 빈약하게 표시됨

**필요한 조치**
- 트레이너 등록 폼에 소개글, 전문 분야 태그, 자격증, 프로그램 항목 추가
- 데이터베이스 구조에 해당 필드 추가

## 7. 예상 개발 범위 요약

| 단계 | 포함 항목 | 목적 |
|------|----------|------|
| **1단계** (긴급) | 항목 1~5 | 보안 사고 방지, 오픈 최소 조건 충족 |
| **2단계** (필수) | 항목 6~10 | 실제 운영 가능한 서비스 수준 |
| **3단계** (개선) | 항목 11~15 | 운영 안정성 및 관리 효율화 |
| **4단계** (고도화) | 항목 16~20 | 서비스 경쟁력 강화 |

> 1단계와 2단계 완료 후 소프트 오픈(제한된 회원 대상 시범 운영)을 권장합니다.
