## 3. 긴급 — 오픈 불가 수준

> 아래 항목이 해결되지 않으면 서비스 오픈 시 보안 사고 또는 데이터 사고가 발생할 수 있습니다.

---

### 3-1. AWS 자격증명 코드 내 평문 저장

**현황**
`backend/.env` 파일에 실제 AWS Access Key와 Secret Key가 평문으로 저장되어 있습니다.

**위험**
- 해당 파일이 GitHub 등 외부 저장소에 업로드될 경우 AWS 계정 전체가 탈취될 수 있습니다.
- 탈취된 키로 AWS 리소스 무단 사용 시 수백만 원 이상의 비용이 발생할 수 있습니다.

**필요한 조치**
- `.env` 파일을 `.gitignore`에 반드시 추가
- Lambda 환경에서는 AWS IAM Role 사용 (별도 키 불필요)
- 로컬 개발 환경에서는 AWS CLI 프로파일 방식 사용
- 현재 사용 중인 Access Key는 즉시 비활성화 후 재발급

---

### 3-2. JWT Secret 기본값 사용

**현황**
`backend/.env`의 `JWT_SECRET`이 `'your-secret-key'` 또는 이와 유사한 기본값으로 설정되어 있습니다. 해당 값이 소스 코드에 노출되거나 추측 가능한 값이면 토큰을 임의로 위조할 수 있습니다.

**위험**
- 공격자가 관리자 JWT를 직접 서명하여 모든 관리자 API를 무단 호출 가능
- 회원 JWT 위조로 타인의 예약 조회 가능

**필요한 조치**
- 최소 32바이트 이상의 무작위 문자열로 교체 (`openssl rand -base64 32`)
- `.env`에만 저장, 코드에 하드코딩 금지
- 운영 서버에서는 AWS Secrets Manager 또는 환경변수로 주입

---

### 3-3. CORS 전체 도메인 허용

**현황**
```
모든 출처(도메인)에서 API 호출 허용 상태
```

**위험**
- 어떤 외부 사이트에서도 이 서비스의 API를 직접 호출할 수 있습니다.
- 자동화 스크립트로 대량 예약 생성 또는 데이터 수집이 가능합니다.

**필요한 조치**
- 실제 서비스 도메인만 허용하도록 CORS 설정 변경
- 예: `https://gympt.co.kr`만 허용

---

### 3-4. 동시 예약 시 중복 발생 가능 (레이스 컨디션)

**현황**
예약 생성 시 "이 시간에 이미 예약 있는지 확인 → 없으면 저장" 순서로 처리합니다.
두 명이 동시에 같은 시간을 예약하면 둘 다 "확인" 통과 후 동시에 저장될 수 있습니다.

**위험**
- 같은 트레이너·날짜·시간에 2명 이상이 예약되는 중복 예약 발생

**필요한 조치**
- DynamoDB 조건부 쓰기(Conditional Write) 적용
- "저장 시점"에 중복 여부를 원자적으로 검증하도록 변경

---

### 3-5. 전화번호·날짜 유효성 검증 없음

**현황**
- 전화번호: 공백·하이픈만 제거하고 자리수 검증이 없어 `1`, `abc` 같은 값도 저장됩니다.
- 날짜: 과거 날짜나 10년 후 날짜도 예약이 가능합니다.

**필요한 조치**
- 전화번호: 10~11자리 숫자 형식 검증 추가
- 날짜: 오늘 이후 ~ 최대 3개월 이내만 선택 가능하도록 제한

## 4. 높음 — 오픈 전 필수 개발

> 서비스의 기본적인 신뢰성과 사용성을 위해 오픈 전 개발을 강력히 권장합니다.

---

### 4-1. 예약 확정 알림 없음

**현황**
예약이 완료되어도 회원에게 어떠한 외부 연락도 가지 않습니다. 인앱 알림 기록은 저장되나 실제 문자나 카카오 알림톡은 발송되지 않습니다.

**영향**
- 회원이 예약 날짜·시간을 잊어버려 노쇼 발생 가능성 높음
- 예약 신뢰도 하락

**필요한 조치**
- 예약 완료 시 카카오 알림톡 또는 문자(SMS) 발송
- 발송 내용: 트레이너명, 날짜, 시간, PT 유형
- (선택) 하루 전 예약 리마인드 알림 발송

---

### 4-2. 회원이 직접 예약 취소 불가

**현황**
예약 취소는 관리자 페이지에서만 처리 가능합니다. 회원 대시보드에서는 조회만 가능합니다.

**영향**
- 영업시간 외 취소 요청 처리 불가
- 직원 응대 부담 증가

**필요한 조치**
- 회원 대시보드에서 예약 취소 기능 추가
- 취소 가능 기준 시간 설정 (예: 수업 2시간 전까지만 취소 가능)

---

### 4-3. 결제 기능 없음

**현황**
PT 패스 구매 페이지가 있으나 실제 결제 수단이 없습니다. 패스 발급은 관리자가 수동으로 처리해야 합니다.

**필요한 조치** (선택 사항, 비즈니스 정책에 따라 결정)
- 온라인 결제 연동: 토스페이먼츠, 아임포트(포트원) 등
- 결제 완료 시 자동으로 패스 발급
- 또는 현장 결제 후 관리자 수동 발급 안내 문구 강화

---

### 4-4. 예약 이력 기록 없음

**현황**
예약을 취소하면 데이터베이스에서 완전히 삭제됩니다. 누가, 언제, 어떤 이유로 취소했는지 기록이 남지 않습니다.

**영향**
- 회원과 분쟁 시 근거 자료 없음
- 취소율, 노쇼율 통계 산출 불가

**필요한 조치**
- 예약 삭제 방식 → 상태를 'Cancelled'로 변경하는 방식으로 전환
- 변경 일시(`updatedAt`), 변경 사유(`cancelReason`) 필드 추가

---

### 4-5. 회원 비밀번호 재설정 불가

**현황**
회원이 비밀번호를 분실한 경우 재설정 수단이 없습니다. 개발자에게 DB를 직접 수정 요청해야 합니다.

**위험**
- 비밀번호 분실 시 계정 접근 불가 → 회원 이탈
- 관리자가 DB를 직접 수정하는 운영 리스크

**필요한 조치**
- 문자 인증(SMS OTP)을 통한 비밀번호 재설정 플로우 구현
- 또는 관리자 페이지에서 비밀번호 초기화 기능 제공 (임시 비밀번호 문자 발송)

---

### 4-6. 상담 문의 스팸 방지 없음

**현황**
`POST /stores/:storeId/inquiries`는 인증 없이 누구나 호출 가능하며, 호출 횟수 제한이 없습니다.

**위험**
- 자동화 스크립트로 수천 건의 스팸 문의 생성 가능
- DynamoDB 쓰기 과금 발생

**필요한 조치**
- IP 기반 rate limit 적용 (예: 동일 IP에서 10분에 3건 제한)
- reCAPTCHA 또는 honeypot 필드 추가

---

## 5. 중간 — 운영 품질 개선

> 오픈 후 운영하면서 순차적으로 개선이 필요한 항목입니다.

---

### 5-1. 로고·CMS 이미지 data URL 크기 문제

**현황**
스토어 설정에서 로고 이미지를 업로드하면 S3 Presigned URL을 통해 S3에 저장됩니다. 그러나 현재 DB seed 데모 데이터는 SVG data URL을 DynamoDB 레코드에 직접 저장하고 있습니다.

**위험**
- 실 운영 시 사용자가 큰 이미지를 직접 Base64로 붙여 넣으면 DynamoDB 아이템 크기 제한(400KB)에 걸릴 수 있음
- S3 업로드 경로를 우회하여 대용량 데이터를 DB에 직접 저장 가능

**필요한 조치**
- 백엔드 PUT /stores 에서 `logo.image` 값이 data URL인 경우 400 반환
- 이미지는 반드시 S3 URL(`https://` 형식)만 허용하도록 검증 추가

---

### 5-2. 외부 이미지 CDN 의존 (Unsplash)

**현황**
DB seed 데이터의 히어로·about 섹션·프로그램 카드 이미지가 Unsplash URL을 사용합니다.

**위험**
- Unsplash 정책 변경 또는 URL 만료 시 이미지 깨짐
- 상업적 이용 시 라이선스 확인 필요

**필요한 조치**
- 서비스 오픈 전 실제 헬스장 사진으로 교체
- 이미지는 S3에 업로드하여 자체 CDN(CloudFront) URL 사용

---

### 5-3. 오류 알림 방식 개선

**현황**
일부 관리자 페이지에서 오류 발생 시 브라우저 기본 `alert()` 팝업이 표시됩니다.

**필요한 조치**
- 화면 내 토스트 메시지 또는 인라인 오류 메시지로 교체
- 오류 유형에 따른 안내 문구 세분화

---

### 5-4. 예약 완료 페이지 새로고침 시 데이터 소실

**현황**
예약 완료 후 이동하는 `/complete` 페이지는 이전 페이지의 상태 데이터에 의존합니다. 브라우저 새로고침 또는 직접 URL 접근 시 예약 정보가 표시되지 않습니다.

**필요한 조치**
- 예약 ID를 URL 파라미터로 전달하여 페이지 진입 시 서버에서 직접 조회

---

### 5-5. 서버 로깅 및 모니터링 부재

**현황**
서버에서 `console.error()`만 사용 중입니다. 운영 환경에서 오류가 발생해도 실시간 확인이 어렵습니다.

**필요한 조치**
- AWS CloudWatch 연동 (Lambda 사용 시 자동 수집 가능)
- 주요 API 호출 로그 저장 (언제, 어떤 요청, 성공/실패)
- 오류 발생 시 담당자 이메일·슬랙 알림 설정

---

### 5-6. 등록된 트레이너 상세 정보 불일치

**현황**
관리자 페이지에서 신규 등록한 트레이너는 이름·전문분야·경력·가격·이미지만 저장됩니다. 소개글, 자격증, 운영 프로그램 등 상세 프로필 항목이 없습니다.

**영향**
- 신규 등록 트레이너의 상세 프로필 페이지가 빈약하게 표시됨

**필요한 조치**
- 트레이너 등록/수정 폼에 소개글, 자격증, 프로그램 항목 추가
