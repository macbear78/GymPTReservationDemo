## 1. 예약 상태 정의

| 상태 코드 | 화면 표시 | 색상 | 설명 |
|-----------|----------|------|------|
| `PENDING` | 승인 대기 | 노란색 | 회원이 신청 후 관리자 승인 대기 중 |
| `BOOKED` | 확정 | 초록색 | 관리자가 승인 완료, 예약 확정 |
| `COMPLETED` | 완료 | 파란색 | PT 수업 완료 |
| `CAD` | 취소 | 빨간색 | 회원 또는 관리자가 취소 |
| `NO_SHOW` | 노쇼 | 회색 | 회원이 예약 시간에 나타나지 않음 |

## 2. 전체 예약 흐름 (회원 → 관리자)

```mermaid
sequenceDiagram
    actor 회원
    participant 회원앱 as 회원 앱
    participant API as Backend API
    participant DB as DynamoDB
    actor 관리자
    participant 관리자앱 as 관리자 대시보드

    회원->>회원앱: 날짜·트레이너·시간 선택
    회원앱->>API: GET /available?date=&trainerId=
    API->>DB: PENDING·BOOKED 슬롯 조회
    DB-->>API: 예약된 슬롯 목록
    API-->>회원앱: 예약 가능 시간 반환

    회원->>회원앱: 예약 신청
    회원앱->>API: POST /reservations
    API->>DB: status=PENDING 으로 저장
    DB-->>API: 저장 완료
    API-->>회원앱: { status: "PENDING" }
    회원앱-->>회원: "예약 신청 완료 (승인 대기 중)"

    관리자->>관리자앱: 예약 목록 확인
    관리자앱->>API: GET /reservations
    API->>DB: 전체 예약 조회
    DB-->>API: 예약 목록 (PENDING 포함)
    API-->>관리자앱: 예약 목록 반환
    관리자앱-->>관리자: 승인 대기 건 노란색 표시

    관리자->>관리자앱: "확정" 버튼 클릭
    관리자앱->>API: PATCH /reservations/:id { status: "BOOKED" }
    API->>DB: status=BOOKED 로 업데이트 (STORE + USER 레코드)
    API->>DB: RESERVATION_CONFIRMED 알림 저장
    DB-->>API: 완료
    API-->>관리자앱: { status: "BOOKED" }
    관리자앱-->>관리자: 상태 → "확정" (초록색)

    회원->>회원앱: 내 예약 조회
    회원앱->>API: GET /member/reservations (V2)
    API->>DB: 회원 예약 목록 조회
    DB-->>회원앱: 예약 목록 + 알림
    회원앱-->>회원: "확정" 상태 + 알림 "PT 예약이 확정되었습니다"
```

## 3. 회원 예약 신청 흐름

```mermaid
flowchart TD
    A([회원 앱 접속]) --> B[날짜 선택]
    B --> C[트레이너 선택\n선택 안 함 가능]
    C --> D[가용 시간 조회\nGET /available]
    D --> E{예약 가능한\n시간이 있나?}
    E -- 없음 --> F[다른 날짜 선택]
    F --> B
    E -- 있음 --> G[시간 선택]
    G --> H[PT 유형 선택\n30분 / 60분]
    H --> I{패스권 사용?}
    I -- Yes --> J[패스권 선택\n잔여 세션 확인]
    J --> K{잔여 세션\n있음?}
    K -- No --> L[패스권 없이 진행\n또는 구매]
    K -- Yes --> M
    I -- No --> M[예약 정보 확인]
    L --> M
    M --> N[예약 신청\nPOST /reservations]
    N --> O{중복 슬롯\n확인}
    O -- 중복 --> P[이미 예약된 시간\n안내 메시지]
    P --> G
    O -- 가능 --> Q[(DynamoDB 저장\nstatus = PENDING)]
    Q --> R[패스권 세션 차감\n있을 경우]
    R --> S[완료 응답\nstatus: PENDING]
    S --> T[/예약 신청 완료\n승인 대기 안내/]
```

## 4. 관리자 예약 처리 흐름

```mermaid
flowchart TD
    A([관리자 대시보드 접속]) --> B[전체 예약 목록 로드\nGET /v2/reservations]
    B --> C[승인 대기 건 표시\n노란색 배지]

    C --> D{예약 건 선택}

    D --> E[확정]
    D --> F[취소]
    D --> G[완료]
    D --> H[노쇼]

    E --> E1[PATCH status=BOOKED\n관리자 → 회원 확정 알림 발송]
    E1 --> E2[(STORE + USER 레코드\nstatus=BOOKED 업데이트)]
    E2 --> E3[알림 저장\nRESERVATION_CONFIRMED]
    E3 --> E4[회원 앱 알림\n'예약이 확정되었습니다']

    F --> F1[PATCH status=CAD]
    F1 --> F2[(status=CAD 업데이트)]
    F2 --> F3[취소 예약은\n기본 목록에서 숨김]

    G --> G1[PATCH status=COMPLETED]
    G1 --> G2[(status=COMPLETED 업데이트)]

    H --> H1[PATCH status=NO_SHOW]
    H1 --> H2[(status=NO_SHOW 업데이트)]

    F3 --> Z([목록 새로고침])
    E4 --> Z
    G2 --> Z
    H2 --> Z
```

## 5. 회원 예약 취소 흐름

```mermaid
flowchart TD
    A([내 PT 예약 조회]) --> B[전화번호 입력]
    B --> C[V1 예약 조회\nGET /api/reservations?phone=]
    C --> D{memberToken\n있음?}
    D -- Yes --> E[V2 예약 조회\nGET /member/reservations]
    D -- No --> F
    E --> F[예약 목록 병합\nV1 + V2]
    F --> G[예약 목록 표시]

    G --> H{예약 상태}
    H -- PENDING\n승인 대기 --> I[취소 버튼 표시]
    H -- BOOKED\n확정 --> I
    H -- COMPLETED\n완료 --> J[취소 불가]
    H -- CAD\n취소됨 --> J

    I --> K[취소 버튼 클릭]
    K --> L{V2 예약?}
    L -- Yes --> M[POST /member/reservations/:id/cancel\nBearer Token 필요]
    L -- No --> N[DELETE /api/reservations/:id]
    M --> O{취소 조건 검증}
    O -- 과거 날짜 --> P[취소 불가 안내]
    O -- 본인 예약 아님 --> P
    O -- 통과 --> Q[(status=CAD 업데이트)]
    N --> Q
    Q --> R[목록에서 제거\n로컬 상태 갱신]
```

## 6. 상태 전이도

```mermaid
stateDiagram-v2
    [*] --> PENDING : 회원 예약 신청\n(POST /reservations)

    PENDING --> BOOKED : 관리자 확정\n→ 회원에게 알림 발송
    PENDING --> CAD : 회원 취소\n또는 관리자 취소

    BOOKED --> COMPLETED : 관리자 완료 처리
    BOOKED --> NO_SHOW : 관리자 노쇼 처리
    BOOKED --> CAD : 회원 취소\n또는 관리자 취소

    COMPLETED --> [*]
    NO_SHOW --> [*]
    CAD --> [*]

    note right of PENDING
        노란색 뱃지
        관리자 대시보드에서
        '승인 대기' 건으로 강조
    end note

    note right of BOOKED
        초록색 뱃지
        알림: "PT 예약이 확정되었습니다"
    end note
```

## 7. API 엔드포인트 매핑

### 회원 (Member)

| 동작 | Method | Endpoint | 인증 |
|------|--------|----------|------|
| 가용 슬롯 조회 | `GET` | `/api/v2/stores/:storeId/reservations/available?date=&trainerId=` | 없음 |
| 예약 신청 | `POST` | `/api/v2/stores/:storeId/reservations` | 없음 (userId 필요) |
| 내 예약 목록 조회 (V2) | `GET` | `/api/v2/stores/:storeId/member/reservations` | Bearer memberToken |
| 내 예약 목록 조회 (V1) | `GET` | `/api/reservations?phone=` | 없음 |
| 예약 취소 (V2) | `POST` | `/api/v2/stores/:storeId/member/reservations/:id/cancel` | Bearer memberToken |
| 예약 취소 (V1) | `DELETE` | `/api/reservations/:id` | 없음 |

### 관리자 (Admin)

| 동작 | Method | Endpoint | 인증 |
|------|--------|----------|------|
| 전체 예약 목록 | `GET` | `/api/v2/stores/:storeId/reservations` | 없음 (관리자 JWT 권장) |
| 예약 상태 변경 | `PATCH` | `/api/v2/stores/:storeId/reservations/:id` | 없음 (관리자 JWT 권장) |

### 알림 (Notification)

| 동작 | Method | Endpoint | 발생 시점 |
|------|--------|----------|----------|
| 알림 조회 | `GET` | `/api/v2/users/:userId/notifications` | 회원 앱 접속 시 |
| 알림 읽음 처리 | `PATCH` | `/api/v2/users/:userId/notifications/:notifId` | 알림 클릭 시 |
| 알림 발송 (자동) | 내부 | `createNotification()` | 예약 확정 (`PENDING→BOOKED`) 시 |

## 보충 — 슬롯 중복 방지 로직

```
예약 신청 시:
  STORE#{storeId} / RESERVATION#{date}#{time}#{trainerId}
  → 해당 SK에 PENDING 또는 BOOKED 상태 레코드가 있으면 → 409 중복 에러

가용 시간 조회 시:
  PENDING + BOOKED 상태인 슬롯 → availableSlots 에서 제외
  CAD / COMPLETED / NO_SHOW 상태 → 해당 슬롯 재예약 가능
```

## 보충 — DynamoDB 레코드 구조

예약 1건당 **2개의 레코드** 생성 (원자적 트랜잭션):

```
STORE 레코드 (날짜·트레이너 기준 조회용)
  PK: STORE#store_default
  SK: RESERVATION#2026-03-25#09:00#trainer_001
  status: PENDING → BOOKED
  userId, date, time, reservationId, ...

USER 레코드 (회원 이력 조회용)
  PK: USER#abc-123
  SK: RESERVATION#2026-03-25T09:00:00
  status: PENDING → BOOKED (관리자 변경 시 동기화)
  reservationId, date, time, ...
```
