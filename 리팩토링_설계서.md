## 1. 변경 방향 요약

### Before vs After

| 항목 | Before (현재) | After (목표) |
|------|--------------|-------------|
| 타겟 | 멀티 트레이너 헬스장 | 소규모 PT 스튜디오 (1~3명) |
| 예약 흐름 | 트레이너 → 날짜 → 시간 → 정보 입력 | 날짜 → 시간 → 확인 |
| 트레이너 선택 | 필수 | 선택 (optional) |
| 회원권/패스 | 없음 | remainingSessions + expiryDate |
| 알림 시스템 | 없음 | 리마인더·잔여 세션·재구매·비활동 알림 |
| DB 구조 | 개별 테이블 (reservations, trainers) | 단일 테이블 설계 (Single Table Design) |
| 사용자 식별 | 이름 + 전화번호 | userId 기반 |
| 스토어 개념 | 없음 | storeId 기반 멀티 스튜디오 지원 |

## 2. 현재 시스템 문제점 분석

### 2-1. 예약 구조

```
현재 예약 필드:
- id, name, phone, trainer(필수), date, time, pt_type, status, created_at

문제:
- trainer 가 필수값으로 고정 → 소규모 스튜디오에서 불필요한 단계
- 사용자 식별이 name + phone 조합 → userId 없어 이력 추적 어려움
- storeId 없음 → 단일 스튜디오 가정, 확장 불가
- updated_at 없음 → 변경 이력 추적 불가
- 예약 삭제 시 실제 DELETE 처리 → 감사 로그 없음
```

### 2-2. DynamoDB 구조

```
현재:
- reservations 테이블 (단순 flat 구조)
- trainers 테이블

문제:
- 테이블이 분리되어 있어 연관 데이터 조회 시 N+1 문제
- GSI 없음 → 날짜별 조회, 사용자별 조회 시 Full Scan
- 스토어/사용자/패스/알림을 각각 별도 테이블로 추가하면 관리 복잡도 급증
```

### 2-3. 비즈니스 기능 누락

```
- 회원권(패스) 없음 → 잔여 세션 차감 불가
- 알림 시스템 없음 → 재방문 유도, 리텐션 전략 없음
- 결제 이력 없음 → 수익 관리 불가
```

## 3. DynamoDB 단일 테이블 설계

### 3-1. 테이블 기본 구조

**테이블명:** `gympt` (단일 테이블)

| Key | 타입 | 설명 |
|-----|------|------|
| PK | String | Partition Key |
| SK | String | Sort Key |

---

### 3-2. PK / SK 패턴 정의

#### 스토어 (Store)
| PK | SK | 설명 |
|----|----|------|
| `STORE#{storeId}` | `STORE#{storeId}` | 스토어 기본 정보 |
| `STORE#{storeId}` | `TRAINER#{trainerId}` | 스토어 소속 트레이너 |

#### 사용자 (User)
| PK | SK | 설명 |
|----|----|------|
| `STORE#{storeId}` | `USER#{userId}` | 스토어 회원 목록 조회용 |
| `USER#{userId}` | `PROFILE` | 사용자 프로필 |
| `USER#{userId}` | `PASS#{passId}` | 사용자 보유 패스 |

#### 예약 (Reservation)
| PK | SK | 설명 |
|----|----|------|
| `STORE#{storeId}` | `RESERVATION#{date}#{time}#{userId}` | 날짜별 예약 조회용 |
| `USER#{userId}` | `RESERVATION#{datetime}` | 사용자 예약 이력 조회용 |

#### 결제 (Payment)
| PK | SK | 설명 |
|----|----|------|
| `USER#{userId}` | `PAYMENT#{timestamp}` | 사용자 결제 이력 |
| `STORE#{storeId}` | `PAYMENT#{timestamp}` | 스토어 결제 이력 조회용 |

#### 알림 (Notification)
| PK | SK | 설명 |
|----|----|------|
| `USER#{userId}` | `NOTIF#{timestamp}` | 사용자 알림 목록 |

#### 이벤트 로그 (Audit)
| PK | SK | 설명 |
|----|----|------|
| `STORE#{storeId}` | `EVENT#{timestamp}` | 스토어 이벤트 로그 |

---

### 3-3. GSI (Global Secondary Index) 설계

#### GSI-1: PhoneIndex
- **목적:** 전화번호로 사용자 조회 (로그인, 예약 조회)
- **GSI PK:** `phone`
- **GSI SK:** `PK`

#### GSI-2: StatusIndex
- **목적:** 특정 날짜의 모든 BOOKED 예약 조회 (관리자 뷰)
- **GSI PK:** `storeId`
- **GSI SK:** `status#date`

#### GSI-3: NotifTargetIndex
- **목적:** 미읽음 알림 배치 처리 (알림 발송 대상 추출)
- **GSI PK:** `notifType`
- **GSI SK:** `isRead#createdAt`

---

### 3-4. 접근 패턴 (Access Patterns)

| # | 시나리오 | 조회 방식 |
|---|----------|----------|
| 1 | 특정 날짜의 스토어 전체 예약 조회 | PK=`STORE#{storeId}`, SK begins_with `RESERVATION#{date}` |
| 2 | 사용자 예약 이력 전체 조회 | PK=`USER#{userId}`, SK begins_with `RESERVATION#` |
| 3 | 전화번호로 사용자 찾기 | GSI-1: phone = `010XXXXXXXX` |
| 4 | 스토어 전체 회원 조회 | PK=`STORE#{storeId}`, SK begins_with `USER#` |
| 5 | 사용자 보유 패스 조회 | PK=`USER#{userId}`, SK begins_with `PASS#` |
| 6 | 사용자 알림 목록 조회 | PK=`USER#{userId}`, SK begins_with `NOTIF#` |
| 7 | 리마인더 발송 대상 추출 | GSI-3: notifType=`REMINDER`, isRead=`false` |
| 8 | 스토어 결제 이력 조회 | PK=`STORE#{storeId}`, SK begins_with `PAYMENT#` |

### 4-6. Notification (알림)

```json
{
  "PK": "USER#user_abc123",
  "SK": "NOTIF#2026-03-24T09:00:00Z",
  "id": "notif_001",
  "userId": "user_abc123",
  "storeId": "store_001",
  "type": "SESSION_REMINDER",   // 타입 목록 아래 참고
  "message": "내일 오전 9시 PT 수업이 있습니다. 잊지 마세요!",
  "isRead": false,
  "createdAt": "2026-03-24T09:00:00Z",

  // GSI-3 용 필드
  "notifType": "SESSION_REMINDER",
  "readStatus": "UNREAD"
}
```

**알림 타입 정의:**

| type | 발송 조건 | 내용 |
|------|----------|------|
| `SESSION_REMINDER` | 예약 24시간 전 | "내일 {time} PT 수업이 있습니다." |
| `LOW_SESSIONS` | 잔여 세션 2회 이하 | "잔여 세션이 {n}회 남았습니다. 재구매를 고려해보세요." |
| `REPURCHASE_PROMO` | 잔여 세션 0회 또는 만료 7일 전 | "회원권이 곧 만료됩니다. 지금 재구매하면 {n}% 할인!" |
| `INACTIVITY_ALERT` | 마지막 방문 후 14일 경과 | "{name}님, 오랜만이에요. 다시 시작해볼까요?" |

## 5. API 설계

**Base URL:** `/api/v2`

> v1 API(`/api/trainers`, `/api/reservations`)는 하위 호환성을 위해 유지

---

### 5-1. 예약 API

#### 예약 가능 시간 조회
```
GET /api/v2/stores/{storeId}/reservations/available?date=YYYY-MM-DD&trainerId={id}

Response 200:
{
  "date": "2026-03-25",
  "availableSlots": ["09:00", "10:00", "14:00", "15:00"],
  "bookedSlots": ["11:00", "16:00"]
}
```

#### 예약 생성
```
POST /api/v2/stores/{storeId}/reservations

Body:
{
  "userId": "user_abc123",
  "date": "2026-03-25",
  "time": "09:00",
  "trainerId": "trainer_001",  // optional
  "ptType": "60min",           // optional
  "passId": "pass_001"         // optional, 있으면 세션 차감
}

Response 201:
{
  "reservationId": "res_xyz789",
  "status": "BOOKED",
  "datetime": "2026-03-25T09:00:00",
  "remainingSessions": 6      // passId 있을 때만
}
```

#### 예약 목록 조회 (관리자)
```
GET /api/v2/stores/{storeId}/reservations?date=YYYY-MM-DD&status=BOOKED

Response 200:
{
  "reservations": [...],
  "total": 5
}
```

#### 예약 상태 변경
```
PATCH /api/v2/stores/{storeId}/reservations/{reservationId}

Body:
{
  "status": "COMPLETED"   // BOOKED | COMPLETED | CAD | NO_SHOW
}
```

---

### 5-2. 사용자 API

#### 전화번호로 사용자 조회 또는 생성
```
POST /api/v2/stores/{storeId}/users/lookup

Body:
{
  "phone": "01012345678",
  "name": "홍길동"         // 신규 사용자일 때만 필요
}

Response 200:
{
  "userId": "user_abc123",
  "name": "홍길동",
  "isNew": false,
  "activePasses": [...]
}
```

#### 사용자 예약 이력 조회
```
GET /api/v2/users/{userId}/reservations?limit=20

Response 200:
{
  "reservations": [...],
  "nextToken": "..."     // 페이지네이션
}
```

### 5-3. 패스 API

#### 패스 발급
```
POST /api/v2/stores/{storeId}/users/{userId}/passes

Body:
{
  "totalSessions": 10,
  "expiryDate": "2026-06-30",
  "paymentId": "pay_001"    // optional
}

Response 201:
{
  "passId": "pass_001",
  "remainingSessions": 10,
  "expiryDate": "2026-06-30"
}
```

#### 패스 조회
```
GET /api/v2/users/{userId}/passes?status=ACTIVE

Response 200:
{
  "passes": [
    {
      "passId": "pass_001",
      "totalSessions": 10,
      "remainingSessions": 7,
      "expiryDate": "2026-06-30",
      "status": "ACTIVE"
    }
  ]
}
```

#### 세션 수동 차감 (관리자)
```
PATCH /api/v2/users/{userId}/passes/{passId}

Body:
{
  "action": "DEDUCT",
  "amount": 1
}
```

---

### 5-4. 알림 API

#### 알림 목록 조회
```
GET /api/v2/users/{userId}/notifications?isRead=false&limit=20

Response 200:
{
  "notifications": [...],
  "unreadCount": 3
}
```

#### 알림 읽음 처리
```
PATCH /api/v2/notifications/{notifId}

Body: { "isRead": true }
```

#### 알림 수동 발송 (관리자)
```
POST /api/v2/stores/{storeId}/notifications/send

Body:
{
  "userId": "user_abc123",
  "type": "REPURCHASE_PROMO",
  "message": "오늘 재등록 시 10% 할인!"   // optional, 없으면 기본 메시지
}
```

## 6. 프론트엔드 변경 사항

### 6-1. 예약 플로우 간소화

**Before:**
```
트레이너 선택 → 날짜 선택 → 시간 선택 → PT 유형 선택 → 이름·전화번호 입력
```

**After:**
```
전화번호 입력 (사용자 조회/생성)
→ 날짜 선택
→ 시간 선택
→ (선택) 트레이너 선택
→ 확인 및 예약
```

---

## 7. 마이그레이션 전략

### 7-1. 단계별 전환 계획

#### Phase 1 — 병렬 운영 (하위 호환성 유지)
- 기존 `/api/reservations`, `/api/trainers` 유지
- 신규 `/api/v2/...` 엔드포인트 추가
- 새 DynamoDB 테이블(`gympt`) 생성, 기존 테이블 유지
- 프론트엔드는 기존 v1 API 계속 사용

#### Phase 2 — 데이터 이전
- 기존 `reservations` 테이블 데이터 → `gympt` 단일 테이블로 변환 스크립트 실행
- 기존 `trainers` 데이터 → `gympt` 단일 테이블로 이전
- 이전 완료 후 프론트엔드 API 호출을 v2로 전환

#### Phase 3 — 구버전 정리
- v1 API deprecated 처리 (응답 헤더에 `Deprecation` 추가)
- 기존 테이블 백업 후 삭제

### 7-2. 기존 데이터 변환 매핑

| 기존 필드 | 신규 필드 | 처리 방법 |
|----------|----------|----------|
| `id` | `reservationId` | 그대로 유지 |
| `name` + `phone` | `userId` | phone으로 User 레코드 생성 후 userId 발급 |
| `trainer` (name) | `trainerId` | trainer name → trainer id 매핑 |
| `date` + `time` | `datetime` | 합산하여 ISO 형식으로 변환 |
| `pt_type` | `ptType` | camelCase 변환 |
| `status: 'Confirmed'` | `status: 'BOOKED'` | 상수 변환 |
| `status: 'Cancelled'` | `status: 'CAD'` | 상수 변환 |
| `status: 'No-show'` | `status: 'NO_SHOW'` | 상수 변환 |
| `status: 'Completed'` | `status: 'COMPLETED'` | 상수 변환 |
| `created_at` | `createdAt` | camelCase 변환 |
| - | `storeId` | 단일 스튜디오이므로 `store_default`로 일괄 설정 |
| - | `updatedAt` | created_at과 동일값으로 초기화 |
